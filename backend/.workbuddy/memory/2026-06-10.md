## D 同学 ServiceImpl 实现（优惠券+猫咪+评价）

### 需求审查
审查了 D 同学的任务说明书，发现以下偏差并修正：

| 偏差 | 修正 |
|------|------|
| CatProfiles 表无 store_id 字段 → "按 storeId 查猫咪"不可行 | listCats 暂返回全部猫咪，加 TODO 注释提示加列 |
| Reservations 表无 has_review 字段 | 通过 Reviews 表 countByExample 反查 |
| ReviewSubmitDTO.orderId 是 String 格式 | 解析 "ORD" 前缀提取 Long 型 reservationId |
| CouponUsageMapper 非必要 | UserCoupons 表自带 usedAt/usedReservationId，无需额外关联表 |

### 实现文件

**CouponServiceImpl**（4 个方法）:
- G-1 listCoupons: UserCouponsMapper 按 userId 查 → 联查 Promotions → 组装 CouponVO
- G-2 listAvailableCoupons: 复用 G-1 + 内存过滤 status=unused + minAmount 门槛 + 计算 saving
- G-3 getPromotionRules: 查 isActive=true 的 Promotions + stackingRules 写死
- G-4 calculatePromotion: 先应用折扣券再取最优满减券，遵守叠加规则

**CatServiceImpl**（2 个方法）:
- I-1 listCats: 查全部 CatProfiles → 组装列表 CatVO（imageUrl 拼接，personality 逗号分割，age 计算）
- I-2 getCatDetail: CatProfiles + CatHealthRecords 联查 → 体重历史/疫苗/互动记录/健康评分/建议

**ReviewServiceImpl**（1 个方法）:
- M-1 submitReview: orderId 解析 → 订单归属+状态+重复校验 → INSERT reviews → PointsLog + MemberExt 积分更新

### 关键设计决策
- Promotions.ruleJson 是 PGobject 类型（Object），用 ObjectMapper 安全解析
- 健康评分：体重偏离 40 分 + 疫苗过期 60 分（100 分制）
- 理想体重按品种硬编码（布偶/英短/美短/橘猫/暹罗/缅因）
- 疫苗状态：过期 30 天以上→expired，30 天内→expiring
- 优惠试算需与前端配合：前端传入的 couponIds 是 CouponVO.id（String 型）

## 微信支付沙箱（模拟版）实现

| 文件 | 改动 |

### 三个接口
| 编号 | Method | URL | Token | 说明 |
|------|--------|-----|-------|------|
| P-1 | POST | /api/pay/create?orderId=ORD0000000001 | ✅ | 发起支付 |
| P-2 | POST | /api/pay/simulate?orderId=ORD0000000001 | ❌ | 模拟支付完成 |
| P-3 | GET | /api/pay/status?orderId=ORD0000000001 | ❌ | 查询支付状态 |

### 关键设计
- 模拟方式：不调真实微信 API，直接操作 payments 表
- transactionId 格式：`WX_MOCK_` + 时间戳
- simulatePay 自动更新 reservations.status → "confirmed"
- createPayment 幂等：已有 pending 记录直接返回，已 paid 抛错
- parseOrderId() 解析 "ORD0000000001" → Long 型 reservationId
- ⚠️ 支付测试需要数据库里有订单（reservations 表），C同学 OrderServiceImpl 未完成时可手动 INSERT SQL 测试
